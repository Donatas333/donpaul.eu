# i18n (Internationalization) Workflow Guide

This portfolio site supports multiple languages with a robust translation system. This guide explains how to manage translations efficiently.

## Supported Languages

- **en** - English (default, source language)
- **lt** - Lithuanian
- **nl** - Dutch

## Translation Files Location

All translation files are in `/assets/i18n/`:
- `en.json` - Source of truth (English translations)
- `lt.json` - Lithuanian translations
- `nl.json` - Dutch translations

## How Language Selection Works

1. **Stored Language**: User's last selected language is saved to localStorage (key: `"lang"`)
2. **Browser Detection**: If no saved preference, browser language is detected
3. **Default**: English is used if the browser language isn't supported
4. **Fallback**: If a key is missing in the selected language, English is used as a fallback

## Adding a New Translation Key

### Step 1: Add the key to HTML
Use the `data-i18n` attribute to mark an element for translation:

```html
<!-- For text content -->
<h1 data-i18n="myPage.title">My Title</h1>

<!-- For HTML content -->
<div data-i18n-html="myPage.richContent">Rich content here</div>

<!-- For input placeholders -->
<input data-i18n-placeholder="contact.namePlaceholder" />

<!-- For input values -->
<input data-i18n-value="contact.defaultValue" />
```

### Step 2: Add translation to en.json
Edit `assets/i18n/en.json` and add your key with the English translation:

```json
{
  "myPage": {
    "title": "My English Title"
  }
}
```

### Step 3: Sync with other languages
Run the sync script to add the new key to lt.json and nl.json with a placeholder:

```bash
npm run i18n:sync
```

This will add the following to lt.json and nl.json:
```json
{
  "myPage": {
    "title": "__TODO_TRANSLATE__"
  }
}
```

### Step 4: Translate in other languages
Edit `assets/i18n/lt.json` and `assets/i18n/nl.json` to replace `__TODO_TRANSLATE__` with actual translations:

```json
{
  "myPage": {
    "title": "Mano anglų kalbos pavadinimas"
  }
}
```

## Scanning for Unused Keys

To find missing or unused keys, run:

```bash
npm run i18n:scan
```

This will report:
- ✅ Keys used in HTML but missing from en.json
- 💡 Keys in en.json not used in any HTML

## Supported data-i18n Attributes

| Attribute | Purpose | Example |
|-----------|---------|---------|
| `data-i18n` | Translates text content (textContent) | `<span data-i18n="key.name">Text</span>` |
| `data-i18n-html` | Translates HTML content (innerHTML) | `<div data-i18n-html="key.html">Rich <b>HTML</b></div>` |
| `data-i18n-placeholder` | Translates input placeholder | `<input data-i18n-placeholder="key.holder" />` |
| `data-i18n-value` | Translates input value | `<input data-i18n-value="key.val" />` |

## Global JavaScript Functions

You can control translations programmatically:

```javascript
// Get current language
const currentLang = window.getLanguage();
// Returns: 'en', 'lt', or 'nl'

// Switch to a language
window.setLanguage('lt');
// Automatically applies translations with animation

// Manually apply translations
window.applyTranslations();
```

## Language Switcher Wiring

The site automatically wires up any button with `data-lang` attribute to switch languages:

```html
<button data-lang="en">English</button>
<button data-lang="lt">Lietuvių</button>
<button data-lang="nl">Nederlands</button>
```

When clicked, these buttons call `window.setLanguage()` automatically.

## Best Practices

### ✅ DO

- Keep translation keys short and descriptive: `"nav.home"`, `"button.submit"`
- Use nested objects for organization: `"services.title"`, `"services.description"`
- Run `npm run i18n:sync` after changing en.json
- Run `npm run i18n:scan` before deploying to catch missing keys
- Always provide translations for all keys, don't leave `__TODO_TRANSLATE__` in production

### ❌ DON'T

- Hardcode language-specific text in HTML where a translation key exists
- Edit lt.json or nl.json directly if the key exists in en.json (edit en.json and sync instead)
- Use special characters without escaping in JSON strings
- Remove keys from en.json without checking if they're used

## Workflow Example

**Scenario**: You want to add a new service card with a title and description.

1. Add HTML with translation keys:
```html
<div class="service-card">
  <h3 data-i18n="services.newService.title">New Service</h3>
  <p data-i18n="services.newService.description">Description here</p>
</div>
```

2. Add to en.json:
```json
{
  "services": {
    "newService": {
      "title": "My New Service",
      "description": "This service helps businesses..."
    }
  }
}
```

3. Sync translations:
```bash
npm run i18n:sync
```

4. Translate in lt.json and nl.json:
```json
{
  "services": {
    "newService": {
      "title": "Mano nauja paslauga",
      "description": "Ši paslauga padeda..."
    }
  }
}
```

5. Verify everything is complete:
```bash
npm run i18n:scan
```

Done! ✨

## Troubleshooting

### Language switcher not working
- Check that buttons have `data-lang="en"`, `data-lang="lt"`, or `data-lang="nl"`
- Ensure `assets/js/i18n.js` is loaded (should be before `</body>`)
- Check browser console for errors

### Translations not appearing
- Verify the key exists in en.json (or the selected language's JSON)
- Ensure the HTML element has the correct `data-i18n` attribute
- Check that the attribute key matches the JSON path exactly
- Run `npm run i18n:scan` to check for missing keys

### Some text missing in translation
- Check lt.json and nl.json for `__TODO_TRANSLATE__` entries
- These need to be replaced with actual translations
- The fallback will show English text if a translation is missing
